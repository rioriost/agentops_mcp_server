ゴール別に必要な要素

A. セッション間でのコンテキスト受け渡し
	•	プロジェクト内に “引き継ぎファイル” を置く（例 .agent/handoff.md）
	•	.rules で「開始時に読む」「終了時に更新」を強制
→ .rules はプロジェクト標準の指示を自動でコンテキスト投入できる  ￼

B. “CI/CD的” な修正→build/test→git履歴保存
	•	Agent Panel は ターミナル実行とファイル編集ができる  ￼
	•	Zed の Tasks に verify（build/test/lint）や commit を登録して、同じ流れを何度も回せる  ￼
	•	ツールの自動承認/確認は Tool Permissions で調整可能（安全運用に効く）  ￼

C. エージェントによるテスト生成
	•	Agent に “テストの増分生成” をさせるのが基本（既存テスト実行ログを踏まえて）
	•	さらに強くするなら、MCP で **テスト雛形生成ツール（言語別テンプレ/カバレッジ情報の取得）**を生やす

⸻

推奨アーキテクチャ（現実的に回るやつ）

レベル1：Zed標準だけで回す（すぐ導入）

構成
	•	.rules（プロジェクトの常時ルール）  ￼
	•	tasks.json（verify/test等の Task 定義）  ￼
	•	.agent/handoff.md（引き継ぎの唯一の正）

ワークフロー（人が開始合図→Agentが実行）
	1.	Agent に「handoff を読んで今日の作業を開始」
	2.	Agent が修正案→編集
	3.	Task で verify（build/test）実行
	4.	成功なら git status/diff をまとめて commit（または commit メッセージ生成）
	5.	handoff.md を更新して終了

ここまで全部、Agent Panel の「編集/検索/ターミナル」ツールで成立します。  ￼

⸻

レベル2：MCP を足して “セッション間” を強くする（おすすめ）

Zed は MCP サーバで 追加ツールを提供できます。  ￼
ここで狙うのは「会話ログを手作業でまとめない」「引き継ぎを機械的に整える」「コミット規約を自動化」あたり。

MCP サーバに持たせると効くツール例
	•	handoff_read() / handoff_update(summary, decisions, next_actions)
	•	session_log_append(text)（プロンプト・結果・コマンドログを 1 箇所に集約）
	•	repo_verify()（言語ごとに最適な verify を呼ぶ薄いラッパ）
	•	repo_commit(message, files="auto")（コミット規約に沿って実行）
	•	tests_suggest()（失敗ログと変更差分を材料にテスト案を生成）

メリット
	•	Agent 側の手順が短くなってミスが減る
	•	“引き継ぎフォーマット” や “commit ルール” をツール側で固定できる

⸻

レベル3：外部エージェント（ACP）も組み合わせる（必要なら）

Zed は Zed 標準エージェント以外に、外部エージェント（例: Gemini CLI など）も統合できます。  ￼
「より自律的に長い作業を回したい」「モデルを切り替えたい」場合に選択肢になります（ただしZed内機能との差は出るので用途次第）。
